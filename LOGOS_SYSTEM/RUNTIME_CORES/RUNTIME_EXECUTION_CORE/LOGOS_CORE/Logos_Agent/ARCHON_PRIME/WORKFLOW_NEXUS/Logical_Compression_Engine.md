SYSTEM_SPEC — Logical_Compression_Engine
CANONICAL_TARGET: EMP
ROLE: Formal compression and proof-oriented epistemic condenser
STATUS: Proposed

OBJECTIVE
Build an EMP-side engine that consumes provisional semantic / formal nuggets and compresses them into:
1. compact logical structures
2. proof-candidate scaffolds
3. reusable formal templates
4. promotion-ready epistemic artifact candidates

PURPOSE
This engine is not the raw extractor.
It is the formal condenser that turns multiple related nuggets into fewer, denser, proof-meaningful artifacts.

PRIMARY INPUTS
- semantic nuggets from extractor
- lambda expressions
- PXL forms
- proof fragments
- existing .v artifacts
- unfinished proof traces
- formal mathematical snippets
- symbolic arguments from corpus

CORE FUNCTION
Take many loosely related provisional nuggets and reduce them into:
- one compressed logical family
- one proof candidate
- one artifact cluster suitable for proof compilation

EXAMPLE TRANSFORMATION
Input:
- 12 modal-necessity nuggets
- 4 grounding nuggets
- 3 identity-preservation nuggets

Output:
- one compressed candidate theorem family
- one proof scaffold
- one candidate promotion artifact

COMPRESSION MODES
1. Structural compression
   - merge repeated inference skeletons
2. Formal compression
   - unify equivalent PXL forms
3. Proof compression
   - collapse repeated lemma patterns
4. Artifact compression
   - condense many provisional AA entries into one promotion candidate

PIPELINE
1. ingest EA append streams from EMP backend
2. cluster entries by formal similarity
3. normalize lambda structures
4. normalize PXL structures
5. detect repeated theorem skeletons
6. emit compressed logical artifact
7. build proof scaffold candidate
8. stage for EMP compilation

OUTPUT TYPES
- compressed_logic_family.json
- proof_candidate_bundle.json
- formal_template_registry.json
- theorem_skeleton_candidates.v

FORMAL TEMPLATE EXAMPLES
- empirical_observation → contradiction → modal_necessity
- identity_constraint → grounding_requirement
- impossibility_of_nothingness → necessary_existence

PROMOTION LOGIC
If compressed artifact crosses threshold:
1. EMP compiles
2. if pass, exchange with MSPC for semantic re-read
3. if MSPC semantic re-read also passes
4. send both to Logos_Core
5. Logos_Core may approve for validation / possible promotion path

VALIDATION
- no proof execution inside extractor path
- EMP controls proof compilation exclusively
- fail closed on contradictory merged forms
- preserve provenance links to all contributing AA entries

ARTIFACT MODEL
No new artifact classes.
Compressed outputs live in EMP-owned EAs:
- SMP tracks compression lineage
- AA stores compressed logical bodies

WHY EMP
EMP already handles proof-facing formal material.
Compression belongs here because this is where semantic density becomes proof utility.
