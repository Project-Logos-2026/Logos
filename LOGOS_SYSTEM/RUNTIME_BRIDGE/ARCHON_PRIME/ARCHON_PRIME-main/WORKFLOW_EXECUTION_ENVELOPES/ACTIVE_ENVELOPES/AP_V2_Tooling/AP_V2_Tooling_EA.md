-------------------------------------------------------------------------------
PURPOSE
-------------------------------------------------------------------------------

This artifact defines the operational execution envelope for the
ARCHON_PRIME V2 Tooling system.

The envelope constrains how the tooling pipeline executes, ensuring that
repository mutation, validation, and reporting occur under deterministic
and auditable conditions.

The execution envelope guarantees:

• deterministic tooling execution
• simulation-first enforcement
• artifact routing compliance
• header policy enforcement
• mutation traceability
• failure-safe rollback behavior

The envelope governs all tooling runs invoked through:

WORKFLOW_MUTATION_TOOLING/controllers/pipeline_runner.py

-------------------------------------------------------------------------------
ENVELOPE STRUCTURE
-------------------------------------------------------------------------------

The AP_V2 Tooling execution envelope contains five controlled layers.

Layer 1 — Execution Context
Layer 2 — Simulation Control
Layer 3 — Mutation Governance
Layer 4 — Artifact Routing
Layer 5 — Reporting and Commit Control

All tooling operations must execute within these layers.

-------------------------------------------------------------------------------
LAYER 1 — EXECUTION CONTEXT
-------------------------------------------------------------------------------

Execution context defines the runtime state required for a tooling run.

Context is created by:

runtime_context_manager.py

Required context objects:

ExecutionContext
PipelineConfiguration
TargetRepositoryState
ArtifactRegistryHandle

ExecutionContext fields:

{
  "execution_mode": "simulate | execute",
  "target_repository": "<repo_root>",
  "active_stage": "<pipeline_stage>",
  "simulation_status": "pending | pass | fail",
  "artifact_registry": "<registry_id>",
  "start_timestamp": "<ISO8601>"
}

The execution context persists for the entire pipeline run.

-------------------------------------------------------------------------------
LAYER 2 — SIMULATION CONTROL
-------------------------------------------------------------------------------

All mutation operations must pass simulation verification before
execution is authorized.

Simulation modules:

runtime_simulator.py
import_surface_simulator.py
integration_simulator.py
mutation_simulator.py

Simulation gate:

VG-SIM

Simulation must produce:

simulation_report.json

Required report fields:

{
  "runtime_simulation": "PASS | FAIL",
  "import_surface_simulation": "PASS | FAIL",
  "integration_simulation": "PASS | FAIL",
  "mutation_simulation": "PASS | FAIL"
}

Execution mode authorization rule:

If any simulation returns FAIL:

    execution_mode must remain "simulate"
    mutation operations prohibited

-------------------------------------------------------------------------------
LAYER 3 — MUTATION GOVERNANCE
-------------------------------------------------------------------------------

Mutation operations occur only within the crawl execution stage.

Mutation modules:

header_injector.py
import_rewriter.py
schema_repair.py
dependency_rewriter.py
module_normalizer.py

Mutation governance rules:

MG-001
All mutation operations require patch plan generation.

MG-002
Patch plan must be generated by patch_generator.py.

MG-003
Patch plan must include deterministic transformation steps.

MG-004
All mutation operations must log a MutationRecord.

MutationRecord schema:

{
  "module": "<module_path>",
  "mutation_type": "<header|import|schema|structure>",
  "patch_id": "<uuid>",
  "simulation_verified": true,
  "execution_timestamp": "<ISO8601>"
}

-------------------------------------------------------------------------------
LAYER 4 — ARTIFACT ROUTING
-------------------------------------------------------------------------------

All artifacts produced by the tooling system must route through the
artifact routing subsystem.

Router module:

runtime/artifact_router.py

Artifacts must never be written directly by tooling modules.

Routing rule:

artifact_router.route(artifact, artifact_type, metadata)

Artifact routing targets:

PIPELINE EXECUTION OUTPUT

WORKFLOW_MUTATION_TOOLING/AP_SYSTEM_AUDIT/

TARGET REPOSITORY AUDIT OUTPUT

WORKFLOW_TARGET_AUDITS/reports/

Routing validation rules:

AR-001
All artifacts must include metadata headers.

AR-002
Artifact type must match routing_table.json entry.

AR-003
Invalid routing paths cause pipeline halt.

-------------------------------------------------------------------------------
LAYER 5 — REPORTING AND COMMIT CONTROL
-------------------------------------------------------------------------------

Final pipeline stage produces the execution report suite.

Generated artifacts:

validation_report.json
repair_event_log.json
mutation_log.json
crawl_execution_log.json
crawl_status.json
PIPELINE_EXECUTION_MANIFEST.json

Reporting module:

report_generator.py

Commit authorization rules:

CA-001

Commits may occur only when execution_mode = "execute"

CA-002

Simulation must have passed in current session.

CA-003

Architect authorization must be present.

Commit module:

commit_finalizer.py

-------------------------------------------------------------------------------
FAILURE HANDLING
-------------------------------------------------------------------------------

Pipeline failure triggers one of three failure states.

FAILURE TYPE 1 — Simulation Failure

Action:

Abort mutation execution
Generate simulation_failure_report.json

FAILURE TYPE 2 — Module Mutation Failure

Action:

Invoke repair_engine.py
Retry repair operations
If repair fails → quarantine module

FAILURE TYPE 3 — Routing Failure

Action:

Abort pipeline
Generate routing_violation_report.json

-------------------------------------------------------------------------------
TRACEABILITY REQUIREMENT
-------------------------------------------------------------------------------

All execution runs must produce a traceable execution manifest.

Manifest file:

PIPELINE_EXECUTION_MANIFEST.json

Required fields:

{
  "pipeline_version": "AP_V2",
  "execution_mode": "simulate | execute",
  "stages_executed": [],
  "artifacts_produced": [],
  "mutations_applied": [],
  "simulation_status": "<PASS|FAIL>",
  "completion_timestamp": "<ISO8601>"
}

-------------------------------------------------------------------------------
ENVELOPE GUARANTEE
-------------------------------------------------------------------------------

The AP_V2 Tooling Execution Envelope guarantees that:

• no repository mutation occurs without simulation verification
• all mutations are deterministic and auditable
• all artifacts are routed through the canonical artifact router
• all pipeline runs produce a verifiable execution manifest
